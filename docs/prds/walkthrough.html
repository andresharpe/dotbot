Critical path

What gates everything

PRD-01 / 02→ PRD-04 / 13→ PRD-06 (hooks)→ PRD-12 (semantics)

Four sequential hops. Adding more than 4 agents to a batch doesn't go faster — Batch 3 is the widest at 4 PRDs in flight, which sets the parallelism ceiling. Minimum viable end-to-end = Batches 1–3 alone (9 PRDs) — a runnable system. PRD-08, PRD-09, PRD-10 are polish.

Phase 1 · Foundation · Batch 1

PRD-01Canonical data model

TaskInstance, TaskDefinition, WorkflowRun shapes; 9-status enum with forward + named recovery transitions; date-prefixed per-run layout; schema_version.

Problem

The shape of a task on v4 is implicit and varies across modules. The status enum has redundant values (split) and tasks live in per-status folders that don't reflect time or origin.

Solution highlights

  • Closed core schema with namespaced extensions.<ns> for custom data; unknown top-level fields rejected.
  • 9 statuses: todo → analysing → analysed → in-progress → done plus terminals failed/skipped/cancelled plus needs-input. split dropped.
  • Forward-only state machine with named recovery edges (done→todo reopen, failed→todo retry, skipped→todo unskip). cancelled is terminal — no recovery.
  • IDs: t_/wr_ + 8-char nanoid (canonical in JSON); directory/filename uses the 4-char prefix.
  • Per-run directories under workspace/tasks/workflow-runs/<YYYY-MM-DD-slug-id>/; standalone tasks are single files under workspace/tasks/standalone/.
  • Live status (churns) gitignored at .control/workflow-runs/<wr_id>.json.
Depends on
none — the foundation
Out of scope
concurrency enforcement (PRD-02), worktree (PRD-03), HTTP API (PRD-04), migration (deliberately none)

Phase 1 · Foundation · Batch 1

PRD-02Workflow isolation + concurrency rules

Workflow-level isolated: true|false (default true). Many isolated runs may overlap; at most one non-isolated. Standalone tasks are workflows-of-one.

Problem

Today's isolation is per-task (skip_worktree) with no clear concurrency model. Two workflows may collide on the working tree with no enforcement.

Solution highlights

  • Top-level isolated in workflow.yaml; tasks inherit. Per-task skip_worktree/working_dir/external_repo removed from the framework schema.
  • Default: isolated: true for both standalone tasks and workflows.
  • Concurrency rule (pure function): any isolated run may start; a non-isolated run is blocked by any other non-isolated running run.
  • No-git / zero-commit project: isolated runs refused with a clear error.
  • Rule lives in Dotbot.Workflow; enforcement is PRD-04's job.

none

Phase 2 · Plumbing · Batch 2

PRD-04Runtime HTTP server + project registry

Machine-wide runtime process, bearer-token auth, mutex-per-task/run with ID-ordered acquire, HTTP API for everything the MCP and UI consume.

Problem

v4 has no machine-wide runtime — UI and MCP both call PowerShell modules in-process, each project has its own UI server, no shared registry of which projects exist.

Solution highlights

  • Listener on 127.0.0.1, ephemeral port; bearer token in ~/.dotbot/runtime.json (mode 600).
  • Endpoint discovery cascade: env vars > merged settings > ~/.dotbot/runtime.json.
  • In-memory mutex per task ID + per run ID; multi-target locks acquire in canonical ID order.
  • HTTP surface: /projects, /tasks, /tasks/<id>/status, /workflows/runs, /tasks/next, /tasks/<id>/context.
  • Enforces PRD-02 concurrency rule (409 on non-isolated conflict).
  • Appends events to <project>/.bot/.control/activity.jsonl; UI's file watcher consumes as today.

Phase 2 · Plumbing · Batch 2

PRD-13Workflow registry

Two tiers: built-in workflows at ~/.dotbot/framework/workflows/, project-local at <project>/.bot/workflows/. Project tier wins on name conflict.

Problem

PRDs reference "workflows" and workflow_start throughout, but where workflows live (framework vs project) and how a user adds one isn't defined.

Solution highlights

  • Built-in tier ships with the framework, refreshed by dotbot upgrade.
  • Project tier is committed in the user's project; user-editable.
  • Find-Workflow resolves project tier first; framework as fallback.
  • workflow_list shows both with a source field; same name in both → project wins with override marker.
  • New CLI: dotbot workflow scaffold <name> copies a built-in to the project tier for customisation.

Phase 3 · Behavior · Batch 3

PRD-03Worktree rewrite

One worktree per workflow run; no junctions; no auto-merge. Completion → directory removed, branch stays. Cancellation → wip commit, then removed.

Problem

v4's worktree is per-task with junction setup for 7 shared directories and patch-replay auto-merge onto main. Couples the framework to a specific integration model and makes the worktree code hard to reason about.

Solution highlights

  • Per-WorkflowRun worktree at ../worktrees/<repo>/<YYYY-MM-DD-slug-id>/.
  • Branch workflow/<slug>-<id> (or task/<slug>-<id> for standalone) from main.
  • Parallel tasks within a run share the worktree; depends_on dictates flow.
  • No junctions: runtime resolves paths from WorkflowRun record (worktree_path, project_root).
  • No auto-merge: framework produces a branch, user integrates manually.
  • Cancellation captures uncommitted work as a wip: commit on the branch before removing the directory.
  • New CLI: dotbot prune-branches.

PRD-01 · PRD-02
Out of scope
cross-workflow dependencies, push to remote, long-lived divergence rebases

Phase 3 · Behavior · Batch 3

PRD-05Plugin executors

File-discovered executor plugins under runtime/executors/<name>/. Initial set: prompt, script, mcp.

Problem

v4's dispatch on task type is hard-coded inside Dotbot.Task and the workflow process loop. Adding a new type requires editing multiple files.

Solution highlights

  • Each executor: a folder with metadata.yaml + script.ps1 exporting Invoke-Executor.
  • Metadata declares task_type, required_fields, supports_worktree, supports_analysis, max_executor_duration.
  • Dispatcher in Dotbot.Task picks by task.type; validates required fields before invocation.
  • prompt executor is a shim around the existing Claude harness launch logic — extracted, not rewritten.
  • Child processes register with Dotbot.Process (kept intact) for cancellation cascade in PRD-12.

Phase 3 · Behavior · Batch 3

PRD-06Plugin transition hooks

File-discovered hooks at runtime/hooks/transitions/enter-<status>/. Synchronous with per-hook timeout; failures can abort the transition.

Problem

v4's side effects on status changes (verification on done, worktree setup on in-progress) are buried inside MCP tools and inline in the workflow process. Adding a new side effect means editing the wrong file.

Solution highlights

  • Each hook: folder with metadata.yaml + script.ps1 exporting Invoke-Hook.
  • Metadata: target_statuses, max_duration, abort_on_failure.
  • Shipped: enter-in-progress (worktree ensure), enter-done (verification + commit info + session close), enter-failed (notification + diagnostic archive), enter-skipped, enter-cancelled.
  • Aggregation hook (PRD-12) fires from all four terminal-status hooks.
  • Runs synchronously inside Set-TaskStatus (inside the task mutex); slow hooks are killed at max_duration.
  • abort_on_failure: true reverts the transition on hook failure.

Phase 3 · Behavior · Batch 3

PRD-07MCP tool surface

Ten focused tools, each a thin HTTP wrapper over the runtime. Old task-mark-* surface removed.

Problem

The v4 MCP surface has many overlapping tools, each re-implementing logic that should live in one place. Status transitions are scattered across seven separate tools.

Solution highlights

  • Task tools: task_create, task_get, task_list, task_update, task_set_status, task_get_next, task_get_context.
  • Workflow tools: workflow_start, workflow_get, workflow_list.
  • Each tool's script.ps1 is ≤15 lines: build body → HTTP-call runtime → return.
  • task_set_status's metadata enumerates all 9 statuses + side effects inline.
  • Actor auto-injected as mcp:<session-id> on every mutation.
  • MCP server (~/.dotbot/framework/src/mcp/dotbot-mcp.ps1) resolves project_id from cwd at startup; caches runtime endpoint.

Phase 4 · Integration · Batch 4

PRD-12Workflow runtime semantics

Expansion (manifest → tasks), status aggregation (task transitions → run status), cancellation cascade (run cancel → tasks).

Problem

No PRD specifies exactly what happens between "click start" and the first task running, when WorkflowRun aggregation fires, or what happens to in-flight tasks on cancellation.

Solution highlights

  • Expansion: resolves workflow.yaml via PRD-13, evaluates form modes, materialises TaskInstances with provenance, creates run.json + live-status record. Atomic — all-or-nothing.
  • Aggregation: every terminal-status hook (enter-done/failed/skipped/cancelled) calls Update-WorkflowRunStatus. All required tasks done → run completed; any required failed → run failed.
  • Cancellation cascade: terminates children via Dotbot.Process; transitions non-terminal tasks to cancelled; calls worktree cleanup with wip commit; run → cancelled (direct, no intermediate state).
  • Cancellation grace period: 30s default.

Phase 4 · Clients · Batch 4

PRD-08UI server as HTTP proxy

TaskAPI.psm1 rewritten as a thin proxy. Bearer token never reaches the browser.

Problem

The UI server today owns task mutation directly; the MCP server has parallel logic. Two writers of the same state with subtle differences.

Solution highlights

  • UI server reads runtime endpoint at startup; resolves project_id once via POST /projects.
  • Each /api/tasks/* handler translates to an authenticated runtime call, injecting actor: "ui:<user>".
  • Frontend (browser JS) keeps calling /api/... same-origin — no JS changes.
  • Runtime token never written into a response body, never logged, never echoed in a header.
  • UI server refuses startup if runtime unreachable.
  • Aux UI modules (AetherAPI, ControlAPI, ProductAPI, DecisionAPI, …) untouched.

Phase 4 · Clients · Batch 4

PRD-09Prompts + agents rewrite

Sweep every prompt and agent file to use the new MCP tool names and the forward-only state-machine prose.

Problem

After PRD-07 lands, the old task_mark_* tools don't exist. Today's prompts and agent personas instruct Claude to call those names — the next workflow run would fail at runtime.

Solution highlights

  • Replace every old tool name (task_mark_done → task_set_status with status: "done", etc.).
  • Remove references to task_answer_question, task_approve_split, task_create_bulk, task_get_stats.
  • Rewrite lifecycle prose: forward sequence + named recovery + 3 terminal statuses (failed, skipped, cancelled).
  • Drop skip_worktree references; describe workflow-level isolated.
  • Every call example shows project_id explicitly.
  • Hygiene test: grep finds zero hits of removed tool names anywhere under src/ or .bot/content/.

PRD-10Test reshape

New test files per major module; updated tests for overlapping behaviour; removed tests for the deleted surface.

Problem

Existing tests are tightly bound to the v4 surface — task-mark-* tools, per-status folders, junction setup, patch-replay merge. They'll break en masse when PRDs 01–09 land.

Solution highlights

  • New files: Test-DataModel, Test-Transitions, Test-WorkflowRun, Test-Worktree, Test-Runtime-HTTP, Test-Executors, Test-Hooks, Test-MCPSurface, Test-UIProxy, Test-PromptHygiene.
  • Each test asserts on externally observable behaviour, not implementation details.
  • Each PRD lands its own tests; PRD-10 is the closing tidy — removes dead tests, confirms green across platforms.
  • Layers 1–3 pass on Windows / macOS / Linux. Layer 4 (real Claude API) unchanged.
